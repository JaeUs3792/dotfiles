#+TITLE: fcitx5가 C-M-j / C-M-h를 가로채는 문제
#+DATE: 2026-07-20

* 증상

Emacs에서 =C-M-j= (consult-project-buffer, init-vertico.el:63) 가 전혀 동작하지 않음.
Emacs만의 문제가 아니라 Vivaldi 등 **모든 앱**에서 동일. =C-M-h= 도 마찬가지.

키를 누른 뒤 계속 타이핑하면 fcitx5 후보창이 뜨고 이런 목록이 나타남:

: A-1 j   A-2 jo   A-3 ju   A-4 jut   A-5 jak

- 라벨이 =A-1= 인 이유 → keyboard 애드온의 =변환 키 선택(Choose Modifier)= = =Alt=
- 후보가 5개인 이유 → =Candidates per page(PageSize)= = =5=

* 원인

fcitx5 =keyboard= 애드온의 **단어 힌트(word hint)** 기능 기본 핫키.

| 옵션 (내부명)            | GUI 표시명             | 기본값          |
|-------------------------+------------------------+-----------------|
| =Hint Trigger=          | 힌트 모드 실행         | =Control+Alt+H= |
| =One Time Hint Trigger= | 한 번만 힌트 모드 실행 | =Control+Alt+J= |

fcitx5는 입력기 레이어에서 앱보다 **먼저** 키 이벤트를 받으므로, 이 키들은
Emacs/브라우저까지 아예 도달하지 못하고 소비된다. Emacs 설정 문제가 아니다.

* 진단 방법 (다음에 비슷한 일이 생기면)

1. Emacs 바인딩 자체는 살아 있는지 확인 :: 정상이면 명령 심볼이 나온다.
   #+begin_src emacs-lisp
   (key-binding (kbd "C-M-j"))   ;; => consult-project-buffer
   #+end_src

2. 키가 Emacs까지 **도달은 하는지** 확인 :: 이게 핵심.
   문제의 키를 몇 번 누른 뒤 아래를 평가한다.
   #+begin_src emacs-lisp
   (key-description (substring (recent-keys) -40))
   #+end_src
   - 목록에 해당 키가 보이면 → Emacs 내부 문제 (keymap 우선순위 등)
   - **안 보이면 → Emacs까지 오지도 못한 것. IM(fcitx5)/WM(sxhkd) 레이어를 의심**

3. fcitx5에 등록된 핫키 전체 덤프 :: 설정 파일이 아니라 **런타임 값**을 봐야 한다.
   #+begin_src sh
   for a in keyboard unicode clipboard emoji quickphrase imselector hangul; do
     echo "## $a"
     gdbus call --session --dest org.fcitx.Fcitx5 --object-path /controller \
       --method org.fcitx.Fcitx.Controller1.GetConfig "fcitx://config/addon/$a" \
       | tr ',' '\n' | grep -iE "'(Control|Alt|Super|Shift)[^']*'"
   done
   #+end_src

4. WM 쪽도 배제 :: =grep -n 'ctrl' ~/.config/sxhkd/sxhkdrc=

* 해결 — fcitx5-configtool GUI

#+begin_src sh
fcitx5-configtool
#+end_src

1. 상단 **부가기능(Addons)** 탭 선택
2. 목록에서 **Keyboard** 항목을 찾아 오른쪽 **설정(톱니바퀴) 버튼** 클릭
3. 설정 창에서 아래 두 항목의 키를 **지운다**
   - =힌트 모드 실행= → =Control+Alt+H= 제거
   - =한 번만 힌트 모드 실행= → =Control+Alt+J= 제거
   (키 입력란 옆의 지우기/휴지통 버튼 사용)
4. **확인/적용**

fcitx5 재시작 불필요. 즉시 반영된다.

** 검증

#+begin_src sh
gdbus call --session --dest org.fcitx.Fcitx5 --object-path /controller \
  --method org.fcitx.Fcitx.Controller1.GetConfig "fcitx://config/addon/keyboard" \
  | tr ',' '\n' | grep -i "Trigger': "
#+end_src

* 함정 — 이렇게 하면 안 된다

** 키 목록을 비우는 문법이 다르다 (핵심)

fcitx5의 =List|Key= 타입 옵션은 **값이 있을 때와 비었을 때 표기가 다르다.**

#+begin_example
# 값이 있을 때 → 섹션 + 인덱스
[Hint Trigger]
0=Control+Alt+H

# 빈 목록일 때 → 섹션을 지우고 최상위 스칼라로
Hint Trigger=
#+end_example

아래처럼 쓰면 **"빈 항목 1개짜리 목록"** 으로 파싱돼 무시되고 기본값으로 폴백한다.
직접 편집으로 해제가 안 되는 것처럼 보이는 원인이 이것이다.

#+begin_example
[Hint Trigger]
0=
#+end_example

DBus =SetConfig= 로 빈 문자열(=<''>=)을 넘겨도 같은 이유로 되돌아간다.
=fcitx5-configtool= GUI로 지우면 올바른 형식을 알아서 써주므로 가장 안전하다.

#+begin_quote
=conf/quickphrase.conf= 가 정확히 이 함정에 빠져 있다. =[TriggerKey]= + =0== 로
비워놨지만 실제로는 기본값 =Super+grave= / =Super+semicolon= 이 살아 있다.
고치려면 =TriggerKey== 한 줄로 바꾸거나 GUI에서 지울 것.
#+end_quote

** fcitx5를 재시작하지 말 것 → Emacs가 먹통이 된다

Emacs는 XIM 경로로 fcitx5에 물려 있다.

: XMODIFIERS=@im=fcitx
: x-gtk-use-native-input => nil   ;; GTK 네이티브 입력 미사용 = XIM 사용

XIM은 동기 프로토콜이라 IM 서버가 사라지면 Emacs가 응답을 기다리며 **완전히 멈춘다**.
=fcitx5 -r -d= 를 실행했다가 실제로 Emacs가 먹통이 되어 강제 재시작해야 했다.
GUI 적용은 재시작이 필요 없으므로 이 문제를 피할 수 있다.
